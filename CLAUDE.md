# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Purpose

Personal practice repository for basic LLM integrations — chats, structured data generation, and pandas-based
data manipulation — against both cloud (Gemini, Groq) and local (LM Studio via an OpenAI-compatible endpoint) models.
Each exercise is a small, self-contained module under `src/ai_exercises/exercises/`, not part of a larger application.

## Commands

- Setup: `python3 -m venv .venv && source .venv/bin/activate && pip install -e ".[dev]"`
- Run an exercise: as an installed console script (e.g. `product-revenue`, `reviews-challenge`, `email-summary` —
  see `[project.scripts]` in `pyproject.toml` for the full list), or via `python -m ai_exercises.exercises.<name>`.
- Tests: `pytest` (unit tests only) or `pytest -m integration` (also runs tests needing a live local LLM server).
  Run a single test with `pytest tests/test_product_revenue.py::test_build_products_dataframe_shape`.
- Secrets are read from a `.env` file at the repo root (see `.env.example` for the expected keys:
  `GEMINI_API_KEY`, `LOCAL_LLM_API_KEY`, `GROQ_API_KEY`). Never commit real values.

## Architecture

- **`src/ai_exercises/services/`** wraps each LLM backend behind a small class exposing a consistent pair of
  methods: `chat_completion`/`send_message` for raw calls, and `generate_json(prompt, data)` for structured
  output. `generate_json` always appends strict instructions to the prompt demanding raw JSON (no markdown
  fences, no conversational text) and then strips/parses the response.
  - `GeminiService` uses `google.genai`, supports multi-turn chat via `create_chat()`, and parses JSON with the
    standard `json` module.
  - `LocalLLMService` talks to an LM Studio server (default `http://127.0.0.1:1234/v1`) through the `openai`
    SDK, and parses JSON with `dirtyjson` (more tolerant of malformed output) since local models are less
    reliable at strict JSON.
  - Both are re-exported from `services/__init__.py` for `from ai_exercises.services import GeminiService, LocalLLMService`.
  - These services are intentionally **not** being refactored right now — a future pass will revisit them.
- **`src/ai_exercises/exercises/`** holds one module per exercise. Every module exposes a `main()` (the
  entry point wired up in `[project.scripts]`) and, where there's non-trivial logic beyond calling a service,
  factors it into plain functions (e.g. `parse_reviews`, `build_result_dataframe` in `reviews_challenge.py`;
  `build_products_dataframe`, `filter_by_revenue_and_rating` in `product_revenue.py`) so it can be unit tested
  without hitting a network or a live LLM.
- **`src/ai_exercises/paths.py`** centralizes `RAW_DATA_DIR`/`GENERATED_DATA_DIR`, resolved relative to the
  project root rather than the current working directory — exercises work no matter where they're invoked from.
- **`data/raw/`** holds input fixtures (CSV/txt) consumed by exercises; **`data/generated/`** holds their
  output (gitignored, regenerated by running the scripts).
- **`tests/`** is a pytest suite. Tests that require a live external service (e.g. a running local LLM server)
  are marked `@pytest.mark.integration` and self-skip when the service isn't reachable — see
  `tests/test_local_llm_service.py` for the pattern.
- There is no Groq service class yet — `exercises/groq_chat.py` calls the `groq` SDK directly.

## Adding a new exercise

Follow the existing shape: a module in `src/ai_exercises/exercises/` with a `main()`, reading from
`RAW_DATA_DIR` and writing to `GENERATED_DATA_DIR` via `ai_exercises.paths`, using a service from
`ai_exercises.services` for any LLM calls. Extract non-I/O logic into plain functions so it's testable, add
a matching `tests/test_<name>.py`, and register a console script entry in `pyproject.toml` if it should be
runnable as a standalone command.
